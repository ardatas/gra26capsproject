## Introduction 
Julia sets are fractals generated by iterating a complex quadratic function, defined by the recurrence z_{i+1} = z_i^2 + c, where c is a fixed complex parameter and z_0 is the pixel's coordinate in the complex plane.
z_i+1​=(z_i)^2​+c is split into real and imaginary parts:   
Re(z_i+1​)=Re(z_i​)^2−Im(z_i​)^2+Re(c),
Im(z_i+1​)=2* Re(zi​)*Im(zi​)+Im(c), with escape condition 
∣z_i∣≤2, which equals to Re(z_i)^2 + Im(z_i)^2 <= 4
 In accordance with this final formula, the number of iterations a point takes to diverge (exceed a bounded threshold) determines its color, generating the fractal image.

### Solution Approaches
The scalar version julia_V1 computes one pixel at a time and is used as the reference implementation. The main implementation julia uses SSE to process four pixels at once via __m128 registers. Since pixels can escape at different iteration counts, an active mask keeps track of which of the four lanes are still running, updated each step with _mm_cmple_ps. _mm_movemask_ps is used to stop the loop early, once all four lanes have escaped. Pixels left over when the width isn't divisible by four are handled by reverting to the scalar function.
Two ways of counting escaped pixels were implemented: julia_simd first checks which lanes are active (AND with a mask), then adds 1 to them.  julia/julia_count_optimization instead subtracts the active mask directly from the counter, since an all-ones mask equals -1 in two's complement — doing the same job in one instruction instead of two. A second, independent difference is how the four computed pixels are written to the image: julia_count_optimization stores the four counters to memory and writes each pixel individually (scalar write_pixel, called four times), while julia additionally vectorizes this step (write_pixel4), computing the color channels for all four pixels at once and writing them to memory in a single store.

### Benchmarking Methods
Each parameter of julia was benchmarked independently, holding all others at default values (c=-0.5125+0.5213i, start=-2+1.5i, 800x600, res=0.005, n=100, grayscale), to isolate its individual effect on SIMD speedup: start and c (escape behavior and lane divergence), image resolution and dimensions (overhead amortization), maximum iteration count n (scaling with per-pixel work), color output (per-pixel write cost), and small widths below one SIMD register (remainder-handling overhead). Each measurement is the average of 100 runs via the -B flag.

## Performance and Correctness Analysis
Correctness was verified by comparing every implementation's output against the scalar reference pixel-by-pixel across more than 30 distinct scenarios, covering typical parameter ranges as well as targeted edge cases: image widths from 1 to 800 (systematically covering both remainder and non-remainder cases for the four-wide SIMD lanes), extreme c/start values (c=0, a starting point outside the escape radius, a configuration causing single-lane early escape), zero iterations (n=0). The check-interval variant (K=1,2,4,8,16,1000) was additionally verified to produce identical output to the reference regardless of K, confirming that adjusting the escape-check frequency is a pure performance optimization with no effect on correctness. All test cases passed with byte-identical output.
The SIMD version (julia, V0) is about 3.1x faster than the scalar version (julia_V1) for default settings (4.30ms vs. 13.18ms). But this speedup changes a lot depending on the input. It is highest (~3.6x) when c=0, because then no pixel ever escapes, so all four SIMD lanes do the same amount of work together. It drops to ~2.1-2.2x for values like c=10+10i or c=1+0i, where pixels escape fast but not at the same time — so some lanes finish early and just wait for the slowest one.
Speedup also changes with the max iteration count n: it goes up from 2.1x (n=1) to a peak of ~3.15x around n=64, then goes back down to ~2.2x for very large n, once almost every pixel needs the full iteration count anyway. For image size, speedup keeps going up (2.5x to 3.2x) as the image gets bigger, since the fixed setup cost matters less. For widths below 4 pixels, SIMD doesn't help at all — sometimes it's even slower — because a full group of 4 pixels never forms, but the SIMD setup cost is still paid.
Between the two counting methods, julia_count_optimization (subtract-based) is consistently about 0.5-1% faster than julia_simd (AND-then-add). This difference is small, but it shows up the same way every time we run it, so it's probably real and not just noise.
