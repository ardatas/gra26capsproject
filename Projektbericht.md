## Introduction 
Julia sets are fractals generated by iterating a complex quadratic function, defined by the recurrence z_{i+1} = {z_i}^2 + c, where c is a fixed complex parameter and z_0 is the pixel's coordinate in the complex plane.
z_i+1​=(z_i)^2​+c is split into real and imaginary parts:   
Re(z_{i+1}​)=Re(z_i​)^2−Im(z_i​)^2+Re(c),
Im(z_{i+1}​)=2* Re(z_i​)*Im(zi​)+Im(c), with escape condition 
∣z_i∣>2, which equals to Re(z_i)^2 + Im(z_i)^2 > 4
i denotes the first iteration at which this occurs and determines the pixel's color. If no escape occurs before n, the pixel belongs to the Julia set and is colored black.
### Solution Approaches
The scalar version julia_V1 computes one pixel at a time and serves as the reference. The main implementation julia uses SSE to process four pixels at once via __m128 registers. Since pixels can escape at different iterations, an active mask tracks of which of the four lanes are still running, updated each step with _mm_cmple_ps. _mm_movemask_ps is used to stop the loop early, once all four lanes have escaped. Remaining pixels when the width isn't divisible by four are handled by reverting to the scalar function.
Two ways of counting escaped pixels were implemented: julia_simd first checks which lanes are active (AND with a mask), then adds 1 to them.  julia/julia_count_optimization instead subtracts the active mask directly from the counter, since an all-ones mask equals -1 in two's complement — doing the same job in one instruction instead of two. A second, independent difference is how the four computed pixels are written to the image: julia_count_optimization stores the four counters to memory and writes each pixel individually (scalar write_pixel, called four times), while julia additionally vectorizes this step (write_pixel4), computing the color channels for all four pixels at once and writing them to memory in a single store.To reduce the overhead of checking _mm_movemask_ps every iteration, a check-interval parameter K was introduced: instead of testing the active mask after every step, it is tested only every K iterations, reducing the number of mask checks at the cost of running a few extra iterations on lanes that already escaped.

### Benchmarking Methods
Each parameter of julia was benchmarked independently, holding others at default values (c=-0.5125+0.5213i, start=-2+1.5i, 800x600, res=0.005, n=100, grayscale): start(the top-left coordinate) and c (escape behavior and lane divergence), image resolution and dimensions (overhead amortization), maximum iteration count n (scaling with per-pixel work), color output (per-pixel write cost), and small widths below one SIMD register (remainder-handling overhead). Each measurement averages 100 runs via -B.

### Test Environment
Tested on a system with an AMD Ryzen 9 5950X processor, 3.4 GHZ, 32 GB of Ram, Debian 13.5, 64 Bit using WSL2 under Windows 11 KB5044285. Compiled using GCC 14.2.0 using the options -std=c17 -O3 -g -msse4.2.


## Performance and Correctness Analysis
Correctness was verified via SHA256 hash comparison against the scalar reference across 30+ scenarios: widths 1–800 (covering remainder and non-remainder SIMD cases), extreme c/start values (c=0, a starting point outside the escape radius, single-lane early escape), and n=0. The check-interval variant (K=1,2,4,8,16,1000) produced identical output regardless of K, confirming it's a pure performance optimization. All cases passed byte-identical.
Baseline speedup--Julia is about 3.1× faster than julia_V1 at default settings (4.30 ms vs. 13.18 ms).
Effect of c--Speedup peaks (~3.6×) at c=0, since no pixel escapes and all lanes do equal work. It drops to ~2.1–2.2× for c=10+10i or c=1+0i, where pixels escape fast but not simultaneously — some lanes idle waiting for the slowest.
Effect of n-- Speedup rises from 2.1× (n=1) to a peak of ~3.15× around n=64, then falls to ~2.2× at very large n, once nearly every pixel needs the full count.
Effect of image size--Speedup climbs steadily (2.5× to 3.2×) as the image grows, since fixed setup cost matters proportionally less. Below 4-pixel widths, SIMD gives no benefit — sometimes it's slower — since a full lane group never forms but setup cost is still paid.
Counting method comparison--julia_count_optimization is consistently 0.5–1% faster than julia_simd — small but recurring, likely a real effect rather than noise.
