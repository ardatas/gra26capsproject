# Project Report 

## Introduction

Julia sets are fractals generated by iterating a complex function, defined by the recurrence `z_{i+1} = {z_i}^2 + c`, where `c` is a fixed complex parameter and `z_0` is the pixel's coordinate in the complex plane.

`z_{i+1}=(z_i)^2+c` is split into real and imaginary parts:

- `Re(z_{i+1}​)=Re(z_i​)^2−Im(z_i​)^2+Re(c)`,
- `Im(z_{i+1}​)=2* Re(z_i​)*Im(z_i​)+Im(c)`, with escape condition
  `∣z_i∣>2`, which is equivalent to `Re(z_i)^2 + Im(z_i)^2 > 4`.

`i` denotes the first iteration at which this occurs and determines the pixel's color. If no escape occurs before `n`, the pixel belongs to the Julia set and is colored black.

### Solution Approaches

The scalar version julia_V1 computes one pixel at a time using a simple loop that iterates until either the point escapes or the maximum iteration count n is reached, and serves as the reference implementation for both correctness and performance comparisons.

The main implementation julia uses SSE to process four pixels at once via __m128 registers, computing all four pixels' z values in parallel each iteration. Since pixels can escape at different iteration counts, an active mask tracks which of the four lanes are still running, updated each step with _mm_cmple_ps, so an escaped lane stays permanently inactive. Once all four lanes have escaped, _mm_movemask_ps detects this and the loop exits early. The counter is updated via counts -= active, exploiting that an all-ones mask equals -1 in two's complement to fuse masking and incrementing into one instruction. Remaining pixels, when width isn't divisible by four, fall back to the scalar function.

Pixel output is also vectorized: instead of writing each pixel individually, write_pixel4 computes all four color channels at once using integer SIMD operations and writes them to memory in a single store.

### Benchmarking Methods

Each parameter of `julia` was benchmarked independently, holding others at default values (`c=-0.5125+0.5213i`, `start=-2+1.5i`, `800x600`, `res=0.005`, `n=100`, grayscale): `start`(the top-left coordinate) and `c` (escape behavior and lane divergence), image resolution and dimensions (overhead amortization), maximum iteration count `n` (scaling with per-pixel work), color output (per-pixel write cost), and small widths below one SIMD register (remainder-handling overhead). Each measurement averages 100 runs via `-B`.

### Test Environment

Benchmarks were run on an AMD Ryzen 9 5950X system (3.4 GHz, 32 GB RAM) using 64-bit Debian 13.5 under WSL2 on Windows 11 KB5044285. The program was compiled with GCC 14.2.0 using `-std=c17 -O3 -g -msse4.2`.


## Performance and Correctness Analysis
Correctness was verified via SHA256 hash comparison against the scalar reference across 30+ scenarios: widths 1–800 (covering remainder and non-remainder SIMD cases), extreme c/start values (c=0, a starting point outside the escape radius, single-lane early escape), and n=0. All cases passed byte-identical.

#### Baseline speedup
Julia is about 3.1× faster than julia_V1 at default settings (4.30 ms vs. 13.18 ms).

#### Effect of c
Speedup peaks (~3.6×) at c=0, where many neighboring pixels exhibit similar iteration behavior, reducing SIMD lane divergence. It drops to ~2.1–2.2× for c=10+10i or c=1+0i because neighboring pixels escape at different iterations.

#### Effect of n
Speedup rises from 2.1× (n=1) to a peak of ~3.15× around n=64. At very large n, long-running lanes increasingly determine the runtime of each SIMD group, reducing the benefit of processing four pixels together and lowering the speedup to ~2.2×.

#### Effect of image size
Speedup climbs steadily (2.5× to 3.2×) as the image grows, since fixed setup cost matters proportionally less. Below 4-pixel widths, SIMD gives no benefit — sometimes it's slower — since a full lane group never forms but setup cost is still paid.
